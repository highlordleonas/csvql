## Problem

The TUI keeps small results in memory and writes large results to a session-owned
temporary directory. Normal TUI unmount removes the directory, but the current
contract does not prove file permissions, atomic writes, idempotent cleanup, or
safe recovery after a terminated process. A spill-write failure can also escape
the result handler instead of preserving the prior usable result.

This slice makes temporary result storage explicit, secure, recoverable, and
failure-safe. It also removes long-lived full-query-result ownership from TUI
session state so the result store is the only full-result owner. It does not
solve the earlier full Python materialization that occurs before spill
decisions; that remains v1.1 bounded-result work.

## Goals

- Secure session directories and spill files using operating-system temporary
  storage.
- Prevent partial spill files from becoming valid result handles.
- Make normal cleanup deterministic and safe to repeat.
- Recover abandoned LocalQL spill directories on a later TUI launch when they
  are eligible under the conservative marker-and-lease protocol.
- Keep the TUI usable for small in-memory results when a secure spill workspace
  cannot be created or verified.
- Preserve the previous visible and exportable result if a new result cannot be
  stored and its backing storage remains available.
- Preserve the previous capped preview, while reporting that full-result load,
  export, and save-result are unavailable, if its spill workspace is lost.
- Keep startup recovery and failure reporting bounded and non-destructive.
- Preserve current CLI, Python API, catalog, JSON, exit-code, and normal export
  contracts.

## Non-Goals

- Streaming or bounding DuckDB query execution.
- Retaining failed large results indefinitely in memory.
- Streaming export or recall directly from the spill serialization format.
- Writing temporary results under the project, home directory, or `.csvql`.
- A general-purpose cache, temp-file manager, or process supervisor.
- Cleaning arbitrary temporary directories or legacy directories that lack the
  new marker and lock contract.
- Adding a dependency or changing `uv.lock`.

## Architecture

`TUIResultStore` becomes the sole long-lived owner of full query results. It
owns small in-memory results, spilled results, workspace state, and cleanup, and
it issues and resolves result handles. `TUISessionState` owns only handle
references, bounded display views, History metadata, full-result availability,
buffer-tab metadata, and active-selection metadata. A successful `QueryResult`
must not remain referenced by `last_result`, `_query_results`, or any replacement
long-lived state collection after the result handler returns.

History and buffer selection restore the stored bounded display view without
loading the full result. Export and save-result resolve the active handle
through `TUIResultStore` and may materialize the full stored result for the
duration of the existing operation. Small results remain owned in memory by
the store and use the same handle-based path. This ownership correction bounds
post-handler retention, not DuckDB execution or the initial Python result
materialization.

A small private platform-lock helper in `tui_result_store.py` provides the
minimum POSIX and Windows lease operations; it is not a public extension point.

The store starts with in-memory support available and creates a secure spill
workspace lazily when the first large result needs it. Each spill-capable
session uses a low-level secure temporary directory directly under the
operating system's temporary area and controls its lifecycle explicitly; normal
cleanup must not delegate arbitrary recursive removal to a `TemporaryDirectory`
finalizer. The on-disk v1 grammar is:

- session directory: `localql-tui-v1-{session_id}`, where `session_id` is 32
  lowercase hexadecimal characters from a cryptographically secure random
  source;
- marker: `.localql-session.json` containing exactly `format_version` with
  integer value `1`, the matching `session_id`, and `created_at_utc` as a strict
  RFC 3339 UTC timestamp;
- lease: `.lease`, exactly one byte;
- completed spill: `query-{sequence}.pickle`, where `sequence` is a positive
  base-10 integer without leading zeroes; and
- staging spill: `.query-{sequence}-{token}.tmp`, where `token` is 16 lowercase
  hexadecimal characters from a secure random source.

The directory contains no other entries. The marker contains no query data,
credentials, machine-specific project paths, or additional keys. The session
holds the lease under an exclusive lock for its lifetime, and completed spill
files appear only after atomic replacement.

The current foreign-path rejection remains mandatory. Only paths registered by
the owning store can be deserialized. Handles whose workspace is known to be
lost are invalidated before another load is attempted.

## Permissions And Atomic Writes

Temporary directories and staging files use Python's secure temporary-file
APIs. On POSIX, the implementation verifies owner-only directory and file
permissions. On Windows, it relies on the creating user's temporary-directory
ACL and secure exclusive creation; unsupported permission-bit assertions are
not simulated.

The security boundary protects against accidental deletion, malformed
directories, and other operating-system users where ownership or the user's
temporary-directory ACL provides isolation. It does not claim to defend
against a malicious process already running with the same user identity; such
a process can ordinarily modify that user's files regardless of LocalQL.

Spill writes follow this sequence:

1. Create an owner-only staging file inside the session directory.
2. Serialize the result to the staging file.
3. Close the file successfully.
4. Atomically replace the final spill path in the same directory.
5. Register the final path and return its handle.

If serialization, close, or replacement fails, the staging file is removed and
no handle is registered. Cleanup failure for the staging file is recorded in
the bounded cleanup summary. A process-observed partial write is never treated
as a valid result. The design does not claim durable persistence across power
loss; these files remain disposable session-temporary data.

## Session Lease

The session holds an exclusive lock on one byte of its lease file until cleanup.
POSIX uses nonblocking `fcntl` locking and Windows uses nonblocking `msvcrt`
locking. Lock acquisition failures are treated as uncertainty, never as proof
that a directory is abandoned. Successful acquisition means only that no
cooperating LocalQL process holds the lease at that instant; it is a necessary
eligibility check under this protocol, not proof that no process can access the
directory.

The lock helper owns acquire, probe, release, and close behavior. Cleanup is
safe to call repeatedly even when the lock or directory has already gone.

## Startup Recovery

Before creating a new result store, LocalQL examines only direct operating-
system temporary-directory children with the new LocalQL session prefix. The
scan is best-effort and uses these per-launch operation budgets:

- inspect at most 5,000 direct temporary-root entries;
- validate at most 100 prefix-matching candidates;
- inspect at most 1,024 direct entries inside one candidate;
- read at most 4 KiB from one marker; and
- remove at most 20 abandoned session directories.

Reaching any budget stops that class of work without preventing TUI launch.
The budgets bound LocalQL operations, not operating-system latency for one
filesystem call. If a per-candidate enumeration budget is exhausted before the
candidate is fully enumerated, that candidate is rejected rather than partially
validated.

A candidate is eligible only when all of these checks pass:

1. `lstat` identifies it as a real directory, not a symlink, junction, or other
   reparse point.
2. Its ownership matches the current user where ownership is available. On
   Windows, recovery additionally requires that it remain a direct child of
   the resolved current-user temporary root; the implementation does not
   invent POSIX ownership checks on Windows.
3. Its marker is a direct, non-linked regular file of at most 4 KiB containing
   only the supported format version, session identifier, and UTC creation
   time. The identifier matches the session-directory suffix.
4. Its lease is a direct, non-linked regular file exactly one byte long.
5. Every other direct entry is a non-linked regular file whose complete name
   matches the versioned spill or staging-file grammar. Nested directories,
   unexpected names, links, junctions, reparse points, devices, sockets, and
   other file types reject the whole candidate.
6. Both its marker creation time and directory modification time are at least
   24 hours old; invalid, inconsistent, or future timestamps are rejected.
7. Its lease can be acquired nonblockingly.

The scavenger retains the acquired lease while it repeats the directory,
marker, lease, ownership, timestamp, and complete-entry validation. A changed
identity or any new uncertainty rejects the candidate. Recovery never uses a
recursive tree-removal function. It revalidates and unlinks only the expected
direct regular spill and staging files, then the marker, and finally the lease
and now-empty directory. Platform ordering may require releasing and closing
the lease immediately before unlinking the lease file; no other content is
removed after that release. An unexpected late entry therefore makes the final
directory removal fail safely instead of causing broader deletion.

Normal session cleanup follows the same direct-entry discipline. It removes
only registered spill and staging files plus its validated marker and lease,
then removes the empty workspace. An unexpected entry prevents directory
removal and is reported through the cleanup summary; it is never recursively
deleted.

Concurrent scavengers may observe a held lease or an already-removed path and
skip or treat the latter as success. Active, recent, malformed, unsupported,
foreign-owned, symlinked, reparse-point, over-budget, or otherwise uncertain
candidates are left untouched. Legacy directories without the new marker and
lease are not automatically removed.

Hard kills and power loss can therefore leave files until a later launch or
operating-system cleanup. The public TUI guide must state this plainly.

## Spill Failure Behavior

Startup recovery and secure-workspace creation are best-effort and must not
prevent the TUI from launching. Small results continue to use in-memory storage
when no spill workspace is available. The first large result that requires
spill attempts secure workspace creation and becomes a typed storage error if
that attempt fails; it is never retained as an unbounded memory fallback.

The full query result remains local to the result handler only long enough to
attempt storage and record the resulting handle and bounded view. A previously
valid workspace that disappears or becomes invalid gets one retry in a new
secure workspace for the new result. Disk-full, quota, serialization, and
clearly permanent permission failures are not retried.

Failure behavior depends on the state of the previous result:

| Failure condition | Previous result behavior | New result behavior |
| --- | --- | --- |
| New write fails while previous backing storage remains healthy | Previous view, recall, export, and save-result remain unchanged | History records a sanitized storage error; no handle or tab is created |
| Workspace is lost, but previous result is in memory | Previous view, recall, export, and save-result remain unchanged | Retry once in a new workspace when the new result requires spill |
| Workspace is lost and previous result was spilled there | Previous bounded view remains visible; its handle is invalidated and full-result load, export, and save-result become unavailable with a sanitized explanation | Retry once in a new workspace; success makes the new result active, while failure records a storage error |
| No previous result exists | The no-active-result state remains | History records a sanitized storage error; no handle or tab is created |

A storage failure uses a dedicated state transition that appends a History
error and returns the run to ready without clearing the existing active
selection, result view, or buffer tabs. It must not reuse the ordinary query
error transition, because ordinary query errors currently clear prior result
state. If a lost workspace invalidates the active spilled handle, the state
retains its bounded preview but marks the full result unavailable rather than
claiming it remains exportable.

Selecting History for an invalidated spilled handle still shows its bounded
preview and unavailable status. Export or save-result attempts fail before
starting a worker and show the same sanitized explanation.

The sanitized message may include bounded metadata such as row count and a
capped number of column names, but never row values or a temporary path. The
failed full result is not inserted into any long-lived store or session-state
collection. It becomes eligible for Python garbage collection after handler and
worker references are released; immediate collection timing is not part of the
contract.

For buffer execution, each tabular outcome is stored independently. A storage
failure records an error for that statement, creates no result tab or handle for
it, preserves already stored tabs, and does not prevent later already-computed
outcomes from being processed. Display indexes remain dense across successfully
stored tabular outcomes; a failed storage attempt does not consume an index.
Batch completion returns the run to ready only after all eligible outcomes have
been processed.

## Cleanup Failure Behavior

Normal unmount attempts cleanup once, and repeated cleanup calls remain safe.
Cleanup is best-effort at process shutdown: a deletion failure must not trap the
user in the TUI or convert an otherwise successful session into a crash.

The recovery helper and store return bounded internal cleanup summaries for
tests and diagnostics. The app accumulates counts for validated abandoned
directories whose deletion began but did not complete and for its own workspace
cleanup failures. After a normal `app.run()` return, the launcher emits at most
one sanitized stderr warning containing only the combined count. It emits no
temporary path, file name, query text, or row data, and it never changes the
command's otherwise successful exit code. Successful cleanup is silent.
Unknown, over-budget, or deliberately skipped directories do not produce
warnings. Abrupt termination has no reporting guarantee. This slice does not
add a new logging subsystem.

## Compatibility

- Result handles remain session-local implementation details.
- Successful query, recall, export, and save-result behavior is unchanged while
  its backing storage remains healthy.
- TUI session state stops retaining full query results after handle creation;
  this is an internal ownership correction, not a public API change.
- A new typed internal storage exception may cross from the store to the TUI
  boundary, where it is converted to the dedicated storage-error state
  transition described above.
- No CLI, Python API, catalog, JSON, or exit-code contract changes.
- No data migration is required.

## Test Strategy

Store-level tests cover:

- threshold behavior already supported today;
- TUI launch and small-result behavior when secure spill storage is unavailable;
- safe failure of the first large result when workspace creation fails;
- POSIX directory and spill-file permissions where permission bits apply;
- atomic success and partial-file cleanup for serialization and replacement
  failures;
- active lease protection on POSIX and Windows;
- exact marker schema and size limits;
- regular-file and exact-name enforcement for marker, lease, spill, and staging
  entries;
- recent, malformed, unsupported, symlinked, junction, reparse-point,
  foreign-owned, nested, unexpected-entry, and uncertain candidate rejection;
- identity and content revalidation after lease acquisition;
- removal of an old, valid, unlocked candidate;
- concurrent scavenger behavior;
- temporary-root, candidate, per-candidate-entry, marker-size, and deletion
  budget exhaustion;
- repeated cleanup;
- missing-workspace retry versus non-retryable storage failures; and
- no recursive tree-removal call in either cleanup path.

State-level tests cover:

- handles, bounded views, History metadata, availability, and selection metadata
  without a retained full query result;
- the dedicated storage-error transition preserving active state;
- preview-only state and unavailable full-result actions after handle
  invalidation; and
- dense buffer-tab indexes across successful storage outcomes.

TUI integration tests cover:

- normal unmount cleanup;
- preserving the prior visible and exportable result after storage failure;
- degrading a previous spilled result to preview-only when its workspace is
  lost, while keeping a previous in-memory result fully usable;
- invalidating every handle backed by a lost workspace;
- absence of a long-lived full query result in session state after the
  handler returns;
- sanitized History and status errors;
- the dedicated storage-error transition preserving active result state;
- return to the ready state;
- buffer continuation, preserved earlier tabs, and dense successful-tab indexes
  when one result cannot be stored;
- one post-exit sanitized stderr warning for one or more cleanup failures and
  silence after successful or deliberately skipped cleanup; and
- unchanged successful recall, export, and save-result behavior.

A subprocess test creates a locked session, proves a second process will not
clean it, terminates the owner, ages the fixture deterministically, and proves a
later launch can recover it. Platform-specific lock tests run only on their
corresponding operating systems; the shared behavior contract runs everywhere.
Windows tests explicitly cover reparse-point and junction rejection. Tests
assert absence from long-lived collections rather than immediate garbage-
collection timing.

## Documentation

Update `docs/tui-guide.md` and `docs/ARCHITECTURE.md` to document:

- when result spill occurs;
- secure OS-temporary storage and atomic completion;
- normal cleanup and next-launch recovery;
- the bounded recovery scan and conservative skip behavior;
- the 24-hour recovery threshold and conservative skip rules;
- the possibility that hard-crash files remain until a later launch or OS
  cleanup;
- behavior when a new result cannot be stored; and
- preview-only behavior when the backing workspace for an older spilled result
  is lost.

Documentation must not claim that LocalQL bounds query memory before the v1.1
result-contract work is implemented and verified.

## Rollback

The recovery scan and session lease are internal to the TUI store and can be
disabled without changing user data or public contracts. Rolling back preserves
the existing normal-session temporary-directory cleanup behavior. New marked
directories left by a rolled-back version remain ordinary OS temporary data and
are not interpreted by older versions.

The result-store and session-state ownership changes in checkpoint 1 must roll
back together so state never contains handles that its matching store path
cannot resolve. No persisted user data or migration is involved.

## Implementation Sequencing

The implementation plan must separate two reviewable checkpoints within the
same v1.0.2 milestone:

1. secure lazy workspace creation, atomic writes, sole full-result ownership,
   typed storage failures, and state-preservation behavior; and
2. platform leases, bounded recovery, exact candidate validation, and cleanup
   reporting.

Checkpoint 2 does not begin until checkpoint 1 passes its focused tests. This
sequencing is an implementation and review boundary, not a reduction of the
approved hybrid scope.

## Definition Of Done

This slice is complete only when:

1. Successful full query results are owned only by the result store; session
   state retains handles, bounded views, availability, and metadata but no full
   query result.
2. Secure permissions, same-directory atomic writes, leases, exact candidate
   validation, bounded conservative recovery, and idempotent cleanup pass their
   focused tests.
3. Storage failures preserve the previous fully usable result when its backing
   storage survives and degrade honestly to preview-only when an older spill
   workspace is lost.
4. Failed full results are absent from long-lived store and state collections;
   no immediate garbage-collection claim is made.
5. Small results remain usable when secure spill storage is unavailable.
6. Buffer-result storage failures remain isolated to their statement, preserve
   successful tabs, and do not stop later computed outcomes.
7. Cleanup failures cannot crash shutdown and produce at most one sanitized
   post-exit warning after a normal return.
8. Public documentation matches implemented memory ownership, cleanup, lost-
   workspace, and crash behavior.
9. Both implementation checkpoints pass focused review and the full local
   quality gate passes.
10. Supported operating-system CI proves the shared contract, platform lock
    implementations, and Windows reparse-point rejection before the milestone
    acceptance claim.
11. No dependency, lockfile, CLI, Python API, catalog, JSON, exit-code, or
    healthy-result export regression exists.

This document records the maintainer-approved revised design. It does not
authorize implementation, Git mutation, publication, or release.
