## Stable Contracts

### CLI Contract

The documented CLI command surface is stable for v1. DuckDB owns SQL execution;
CSVQL owns local workflow, table aliasing, source resolution, output rendering,
deterministic errors, and project configuration.

### TUI Contract

`csvql menu` is an optional Textual-powered local terminal workbench. It is
available through the `tui` extra and is not required for the core CLI install.

The TUI starts from the discovered project catalog, one CSV path, or repeated
`--table` mappings. It keeps query history in memory for the current TUI session
only. It writes files only when the user explicitly chooses export, project
catalog save, or derived result source save.

`Ctrl+S` saves the active tabular result as
`.csvql/results/{alias}.csv`, then loads that CSV back into the current Sources
pane with kind `derived`. `Alt+S` is also bound for terminals that emit Alt key
events, and `F11` remains available where it is not intercepted by the OS. macOS
may intercept `F11` for Show Desktop. Derived result sources are explicit
CSV-backed local artifacts, not hidden cache or automatic materialization. The
CSV file persists on disk; the alias is session-local until sources are
explicitly saved to `.csvql.yml`.

### Config Contract

`.csvql.yml` remains strict `version: 1`.

Supported top-level keys:

- `version`
- `tables`

Supported table keys:

- `path`
- `checks`

There is no v1 migration framework, alternate config filename, or new config
schema feature in this release package.

### JSON Contract

The current v0.8 JSON output shapes are the stable v1 runtime contract.
`docs/json-contracts.md` is the detailed JSON authority.

Current shape differences are intentional v1 facts. Query-shaped results,
inspection results, profile results, check results, doctor results, table lists,
and benchmark artifacts are not normalized into a shared envelope for v1.

### Exit-Code Contract

The current exit-code behavior is stable for v1:

- `0`: success, including doctor warnings and zero configured checks
- `1`: general CSVQL error and DuckDB query execution failure
- `4`: missing CSV file
- `6`: invalid table mapping or table alias
- `7`: inspect or sample failure
- `8`: project catalog discovery, parsing, or validation failure
- `9`: saved SQL file failure
- `10`: export path or export format failure
- `11`: configured data-quality checks ran and found failures
- `12`: doctor found concrete project-health failures

### Python API Contract

The v1 Python API is small and project-backed:

- `CSVQLSession.from_config(path)`
- `session.tables()`
- `session.query(sql)`
- `session.run_file(path)`
- `session.inspect(table, exact=False)`
- `session.sample(table, limit=10)`
- `session.profile(table)`
- `session.check(table=None)`
- `session.export(sql_file, out, format="json", force=False)`

The Python API does not provide direct-path sessions, ad hoc table mappings,
config mutation helpers, dataframe helpers, async execution, plugin APIs, a
persistent session-level DuckDB connection, or a second execution engine.

### DuckDB Contract

DuckDB remains the sole SQL engine. The package dependency floor is
`duckdb>=1.5.0,<2`.

This does not make CSVQL a sandbox. DuckDB SQL may access files or external
resources depending on settings and extensions. CSVQL is a local automation tool
for trusted projects.

## Candidate Proof Checklist

Run from a clean worktree on `main`.

Capture baseline truth:

```bash
pwd
git status --short --branch
git log -1 --oneline
```

Run the full local gate:

```bash
uv run ruff format --check .
uv run ruff check .
uv run --all-extras mypy src
uv run --all-extras pytest
```

Run release-readiness proof:

```bash
uv run python scripts/verify_release_readiness.py --work-dir output/release-readiness
```

Run the manual QA gates:

- [Manual v1 QA matrix](../v1-manual-qa.md)
- [TUI QoL QA gate](../tui-qol-qa.md)

Record the manual v1 QA matrix date, commit SHA, terminal app, passed items,
and blockers.

Run the TUI QoL QA gate and record the TUI QoL run id, required terminal
coverage, media artifact paths, passed items, and blockers. Any failed TUI QoL
matrix item blocks `release-candidate eligible`. Any untested or missing-media
TUI QoL item also blocks `release-candidate eligible`.

Run benchmark proof:

```bash
uv run python scripts/benchmark_csvql.py --output-root output/benchmarks
```

Rerunning this command during the final candidate check is preferred.

If a current local benchmark artifact is cited instead of rerunning the
benchmark, it must come from the same candidate-state `HEAD`, and the record
must include both of these paths:

- `output/benchmarks/<run-id>/benchmark.json`
- `output/benchmarks/<run-id>/benchmark-summary.md`

Run an unsupported-claim scan across current authority docs:

```bash
rg -n "v1-ready|v1 ready|production-safe|production ready|production-readiness|production readiness|sandbox-safe|sandbox safety|sandbox|large-file-proven|large file proven|large-file performance|large file performance" README.md CHANGELOG.md docs/development.md docs/PRODUCT_DIRECTION.md docs/ROADMAP.md docs/ARCHITECTURE.md docs/json-contracts.md docs/benchmarking.md docs/failure-gallery.md docs/v1-manual-qa.md docs/tui-qol-qa.md docs/release-readiness.md docs/release-notes/v1.md
```

Matches are expected for guardrails, non-claims, and conditional label rules
and must be classified. A current claim that CSVQL is v1-ready,
production-safe, production-ready, sandbox-safe or sandboxed,
large-file-proven, or broadly large-file performant blocks candidate
eligibility.

## Generated Artifact Policy

Release proof writes local evidence under ignored output directories:

- `output/release-readiness/**`
- `output/benchmarks/**`

Generated wheels, sdists, benchmark JSON, benchmark Markdown summaries, virtual
environments, and scratch transcripts remain local evidence unless a separate
tracked-artifact decision is made.

## Unsupported Claims

The v1 release package does not claim:

- sandbox safety
- safe execution of untrusted SQL
- production readiness
- broad large-file performance
- cloud readiness
- multi-tenant isolation
- a web, dashboard, notebook, AI, dataframe, async, plugin, cache, or
  materialization platform
- persisted TUI query history
- hidden TUI result cache or automatic TUI materialization

## Publish And Tag Boundary

The proof commands are local verification commands only.

They do not:

- publish to PyPI
- push Git tags
- create GitHub releases
- upload artifacts
- mutate external systems
- change the package version

Any final release action requires separate explicit approval even after
candidate status is accepted.

## Final Candidate Decision Template

Use one of these outcomes after the final candidate check:

### `release-candidate eligible`

Use only when all of the following are true:

- release package exists
- full local gate passes
- release-readiness proof passes
- benchmark proof is rerun during the final candidate check, or a current local
  benchmark artifact from the same candidate-state `HEAD` is cited with both
  benchmark artifact paths recorded
- authority docs agree with runtime behavior
- the manual v1 QA matrix is recorded for the candidate state
- the TUI QoL QA gate passes on every required terminal with a recorded run id,
  media artifact paths, and no failed, untested, or missing-media items
- unsupported claims are absent
- generated proof artifacts remain ignored

### `v1-hardening`

Use when the release package exists but proof is stale, incomplete, not run, or
blocked by remaining docs or runtime work.

### `blocked`

Use when a named proof, contract, docs, environment, dependency, or tooling
blocker prevents honest candidate classification.
